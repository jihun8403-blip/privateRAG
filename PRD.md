# PRD — Private RAG System

**버전**: 1.0
**작성일**: 2026-03-11
**상태**: Draft

---

## 1. 제품 개요

### 1.1 제품명

**PrivateRAG** — 주제 중심 자동 수집·검증·검색 시스템

### 1.2 목적

사용자가 정의한 관심 주제를 기반으로 웹에서 자동으로 문서를 수집하고, 관련성을 검증·정제하여 로컬 RAG(Retrieval-Augmented Generation) 인덱스에 저장하는 개인용 지식 관리 시스템.

### 1.3 배경 및 문제 정의

| 문제 | 설명 |
|------|------|
| 정보 과부하 | 관심 주제에 대한 고품질 문서를 수동으로 수집하는 데 과도한 시간 소모 |
| 주제 오염 | 단순 키워드 검색은 의미적으로 무관한 문서를 포함시킴 |
| 모델 비용 | API LLM 단독 사용 시 과도한 토큰 비용 발생 |
| 이력 손실 | 웹 문서는 삭제·수정되므로 원문 보존이 필요 |
| 벤더 종속 | 특정 LLM 또는 검색 API에 종속되면 유연성 저하 |

### 1.4 핵심 가치

- 주제 프로필 기반 정밀 수집 (단순 키워드가 아닌 설명 포함 주제 객체)
- 2단계 관련성 검증 (규칙 기반 + LLM 의미 검증)
- 원문·정제본·RAG 인덱스·이력의 4중 분리 저장
- 모델 후보 풀 + 일일 토큰 예산 + fallback chain
- 온프레미스(로컬) LLM 우선 운용 가능

## 2. 사용자 및 사용 시나리오

### 2.1 주요 사용자

- **개인 연구자·개발자**: 특정 기술 분야 문서를 지속적으로 모니터링하고 질의
- **1인 운용 전제**: 다중 사용자 인증은 MVP 범위 외

### 2.2 핵심 사용 시나리오

#### UC-01. 주제 등록

사용자가 관심 주제를 등록한다. 주제명, 설명, 포함/제외 키워드, 도메인 규칙, 수집 주기, 언어, 우선순위를 설정한다.

#### UC-02. 자동 수집 실행

스케줄러가 주제별 cron 설정에 따라 검색 질의를 생성하고, 검색 API를 호출하여 후보 URL을 수집한다.

#### UC-03. 관련성 검증

수집된 문서에 대해 규칙 기반 1차 필터 후, LLM이 의미적 관련성을 0~1 점수로 평가한다. 기준 미달 문서는 저장하지 않는다.

#### UC-04. 문서 검색 및 질의응답

사용자가 자연어로 질의하면 RAG 파이프라인이 관련 청크를 검색하고 LLM이 답변을 생성한다.

#### UC-05. 이력 조회

동일 URL 문서의 과거 버전(내용 변화)을 조회한다.

#### UC-06. 도메인 룰 관리

URL·도메인에 대한 허용/차단 regex 규칙을 추가·수정·테스트한다.

## 3. 기능 요구사항

### 3.1 주제 관리 (Topic Registry)

| ID | 요구사항 |
|----|----------|
| FR-T01 | 주제를 등록·수정·삭제·비활성화할 수 있어야 한다 |
| FR-T02 | 주제는 이름, 설명, 포함 키워드(must/should), 제외 키워드, 언어, 수집 주기(cron), 우선순위, 활성 여부를 포함해야 한다 |
| FR-T03 | 주제별로 preferred_domain / blocked_domain regex 규칙을 N개 등록할 수 있어야 한다 |
| FR-T04 | 주제 목록을 우선순위 순으로 조회할 수 있어야 한다 |

### 3.2 검색 질의 생성 (Query Planning)

| ID | 요구사항 |
|----|----------|
| FR-Q01 | 주제 프로필을 기반으로 LLM이 복수의 검색 질의를 자동 생성해야 한다 |
| FR-Q02 | 생성 결과는 구조화된 형식(질의문, 의도, 언어)으로 반환되어야 한다 |
| FR-Q03 | 한국어/영어 질의를 모두 생성해야 한다 |
| FR-Q04 | preferred_domain을 활용한 site: 검색어도 포함해야 한다 |
| FR-Q05 | 생성된 질의는 DB에 저장되고 만료일을 가져야 한다 |

### 3.3 수집 실행 (Retrieval Orchestrator)

| ID | 요구사항 |
|----|----------|
| FR-R01 | 검색 API(Brave 등) 호출을 통해 URL 목록을 수집해야 한다 |
| FR-R02 | URL·도메인 regex 룰을 적용하여 차단/우선 분류를 수행해야 한다 |
| FR-R03 | 중복 URL을 제거해야 한다 |
| FR-R04 | URL에서 본문을 추출해야 한다 (trafilatura → readability → playwright 순) |
| FR-R05 | 수집 실행 이력(run_id, 상태, 결과 수)을 기록해야 한다 |
| FR-R06 | 새로운 검색 제공자를 플러그인 형태로 추가할 수 있어야 한다 |

### 3.4 관련성 검증 (Relevance Validator)

| ID | 요구사항 |
|----|----------|
| FR-V01 | 1차: 필수 키워드 포함 여부, 제외 키워드 존재 여부, 문서 최소 길이를 규칙으로 검사해야 한다 |
| FR-V02 | 2차: LLM이 주제 설명과 문서의 의미적 부합 여부를 0~1 점수와 이유로 반환해야 한다 |
| FR-V03 | relevance_score 임계값(기본 0.6)을 주제별로 설정할 수 있어야 한다 |
| FR-V04 | 2차 검증에 사용할 모델은 Model Router가 자동 선택해야 한다 |

### 3.5 문서 저장 및 버전 관리

| ID | 요구사항 |
|----|----------|
| FR-S01 | 원문 HTML/JSON은 파일 시스템에 날짜별 디렉터리로 보관해야 한다 |
| FR-S02 | 정제된 본문, 요약, relevance 메타데이터는 DB에 저장해야 한다 |
| FR-S03 | 동일 URL 문서가 내용 변경 시 신규 버전을 생성하고 이전 버전을 보존해야 한다 |
| FR-S04 | content_hash 기반으로 중복 저장을 방지해야 한다 |

### 3.6 RAG 인덱스

| ID | 요구사항 |
|----|----------|
| FR-I01 | 문서를 청크 단위로 분할하고 토큰 수를 기록해야 한다 |
| FR-I02 | 청크에 대한 임베딩을 생성하고 벡터 저장소에 저장해야 한다 |
| FR-I03 | 임베딩 모델이 변경될 경우 재임베딩 작업을 실행할 수 있어야 한다 |
| FR-I04 | 키워드 검색(FTS5)과 벡터 검색을 하이브리드로 지원해야 한다 |

### 3.7 질의응답

| ID | 요구사항 |
|----|----------|
| FR-A01 | 사용자의 자연어 질의에 대해 관련 청크를 검색하고 LLM 답변을 생성해야 한다 |
| FR-A02 | 답변에 출처(URL, 제목, 수집일)를 포함해야 한다 |
| FR-A03 | 답변 생성 모델을 사용자가 선택할 수 있어야 한다 |

### 3.8 모델 관리 (Model Router)

| ID | 요구사항 |
|----|----------|
| FR-M01 | 모델은 provider, capability_tags, 일일 토큰 예산, 우선순위, fallback 순서를 가져야 한다 |
| FR-M02 | 작업 유형(task_type)과 필요 capability를 기반으로 최적 모델을 자동 선택해야 한다 |
| FR-M03 | 일일 토큰 예산 초과·rate limit·오류 발생 시 다음 후보 모델로 자동 전환해야 한다 |
| FR-M04 | 토큰 사용량은 일 단위(Asia/Seoul 기준)로 초기화되어야 한다 |
| FR-M05 | 모델별 토큰 사용량 및 비용 추정을 기록해야 한다 |

### 3.9 스케줄링

| ID | 요구사항 |
|----|----------|
| FR-SC01 | 주제별 cron 설정에 따라 수집 파이프라인을 자동 실행해야 한다 |
| FR-SC02 | 토큰 사용량 일일 초기화 스케줄이 있어야 한다 |
| FR-SC03 | 오래된 문서의 아카이브 tier 전환(Active→Warm→Cold) 스케줄이 있어야 한다 |

### 3.10 도메인 룰 관리

| ID | 요구사항 |
|----|----------|
| FR-D01 | 4종 룰(blocked_url, blocked_domain, preferred_url, preferred_domain) regex를 CRUD로 관리해야 한다 |
| FR-D02 | 룰 적용 순서는 차단 우선(blocked_url → blocked_domain → preferred_url → preferred_domain)이어야 한다 |
| FR-D03 | 임의 URL에 대해 룰 매칭 테스트 기능을 제공해야 한다 |

## 5. MVP 범위

### 1차 MVP (필수)

- 주제 등록·수정·삭제
- 도메인 regex 룰 CRUD
- LLM 기반 검색 질의 생성
- 검색 API 수집 + URL 필터링
- 원문 저장 + 본문 추출
- 2단계 관련성 검증
- 정제본 저장 + 버전 관리
- 청크 + 임베딩 + 벡터 저장
- 자연어 질의응답
- 모델 후보 풀 + 토큰 예산 + fallback
- APScheduler 기반 자동 실행

### 2차 (후속)

- 과거 버전 비교 UI
- 주제별 수집 성공률·오탐률 대시보드
- Archive tier 전환 자동화
- 모델별 비용 리포트

### 3차 (장기)

- 주제별 자동 브리핑 생성
- 중요도 변화 감지
- 오래된 검색어 자동 재생성

## 6. 제약 사항

- 단일 사용자 환경 (다중 사용자 인증 없음)
- 초기 배포는 Windows 로컬 환경 기준
- 검색 API는 Brave Search API 우선 지원
- 로컬 LLM은 Ollama 호환 API 기준
